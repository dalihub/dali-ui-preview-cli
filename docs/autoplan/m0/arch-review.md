## Verdict: PASS

All ADRs upheld, no drift. M0 implements ADR-001/002/003/007 faithfully; ADR-008 correctly *records* (does not *supersede*) the empirical branch ADR-003 was built to absorb. No finding blocks M1. The two items below are confirmations, not drifts.

## Drift findings

- **none.** No ADR is contradicted by the implementation, and no ADR is left obsolete or in need of a `supersedes`.

The two structurally-interesting points (each resolved in-favour of the existing design, hence *not* drift):

- **A1 (confirmation, not drift): ADR-008 vs ADR-003 ‚Äî no supersedes needed.** ADR-003 deliberately committed only the *structure* ‚Äî "property-reconstructed walk is the guaranteed default; `DumpTree` is optional enrichment gated behind a runtime probe; M1's schema is byte-identical in shape either way" ‚Äî and explicitly deferred the empirical yes/no to ADR-008 (architecture.md OPEN_QUESTIONS; ADR-003 ¬ßConsequences "Neutral"). ADR-008 records the observed branch ("`DumpTree` WORKS headless without D-Bus", 1156-char per-node JSON; the D-Bus/AT-SPI errors are the *external* bridge, irrelevant to local `DumpTree`). Because ADR-003 was written to be true under *both* answers, the "yes" answer **confirms** ADR-003's `DumpTree`-available branch rather than invalidating any decision in it. ADR-008's own Status line states this precisely ("does NOT supersede ADR-003 ‚Äî it confirms ADR-003's 'DumpTree-available' branch is the reality"), and it keeps the property-walk as the Inv-2 floor. This is correct append-only ADR hygiene: structure-decision (ADR-003) stays accepted; result-record (ADR-008) is accepted alongside it. **No `supersedes`, no rewrite of ADR-003 required.** (If anything, a one-line forward-link could be added to ADR-003 pointing at ADR-008, but architecture.md already carries that pointer in its ADR index and OPEN_QUESTIONS, so it is not necessary.)

- **A2 (note, not drift): M0's emitted tree is *not yet* the ADR-003 walk ‚Äî and that is exactly what M0 scoped.** The M0 tree comes verbatim from the vendored harness `CollectActorMetadata` (`server/...template:119‚Äì207`): `GetTypeName()` + `GetCurrentProperty<>()` for type/name/pos/size, with the **hand-rolled `parentOrigin`/`anchor` bounds math** (lines 133‚Äì135) that ADR-003/F1.2 says to *replace* with `CalculateCurrentScreenExtents`. M0/spec.md ¬ß2 explicitly defers that replacement to M1 ("M0 keeps the existing bounds as-is"). So the *floor source* ADR-003 names (property walk, bridge-independent) is already the live source in M0; only the bounds-accuracy upgrade is pending, on schedule. Not drift ‚Äî deferred per the frozen scope.

## Invariant audit

(architecture.md "Key invariants". M0 is the build-infra + spike milestone, so several invariants are M1/M2/M4 concerns and correctly *not yet applicable*. Each below states upheld / violated(file:line) / deferred.)

- **Inv-1 (one source for ids ‚Üî marks):** **deferred ‚Äî not applicable in M0.** Stable ids and overlay marks are an M2/ADR-004 concern; M0 emits no `id`/`mark` (spec.md ¬ß2 "No `id`/`mark` emission in M0"), and there is no overlay renderer. There is consequently no second id-generator to violate the invariant. To be audited when M2 lands the single-DFS id+mark emission in the harness.

- **Inv-2 (tree-source independent of the a11y bridge):** **upheld.** The only tree the CLI emits in M0 is produced by the property walk `CollectActorMetadata` ‚Üí `ExportSceneMetadata` (`server/...template:119, 209, 304`); there is **no** `Accessibility`/`DumpTree` call in the rendered harness (`grep` over the template confirms `DumpTree`/`Accessib` appear nowhere in it). The render runs green with the AT-SPI bridge provably DOWN headless (exec-validation WU-4 "D-Bus/AT-SPI is DOWN ‚Ä¶ render proceeds fine"; spike Evidence A). So a complete, schema-valid tree is produced with no D-Bus dependency ‚Äî exactly the invariant. ADR-008's "`DumpTree` works" result does **not** weaken this: ADR-008 keeps the property walk as the guaranteed floor and only *adds* `DumpTree` as an M1 enrichment, so Inv-2 remains the contract M1 must preserve. The `semanticsSource` metadata field the invariant references is itself an M1 addition (the M0 harness does not emit it yet) ‚Äî consistent with the schema being M1's deliverable.

- **Inv-3 (byte-identical tree across runs):** **deferred ‚Äî not applicable in M0 / not yet enforced.** M0's minimal tree carries floating-point `x/y/w/h` and `opacity`/color from the property walk, has no address-stripping/sorted-children guarantee asserted, and the verbatim ADR test (`render twice | diff` empty) is an M1/F1.4 tier-2 check not run in M0 (M0's F0.4 assertion only checks JSON shape: root-type + non-empty children). Determinism is not yet a frozen M0 requirement (spec.md ¬ß2 defers the canonical schema to M1). No violation observed, but the invariant is not yet in force; audit it at M1 when the canonical emitter (sorted children, stripped addresses, structural ids) is built.

- **Inv-4 (render happens in the fixed image, never the host):** **upheld.** The sole render path is `cli.runRender` ‚Üí `harnessTemplater` ‚Üí `dockerRunner.renderInContainer`, which `docker run --rm`s `ghcr.io/lwc0917/dali-preview-runtime:latest` (`dockerRunner.ts:37‚Äì38, 196‚Äì209`). There is **no** native-DALi link, no host compile/render path: `cli.ts` imports only `fs`/`path` + the three orchestration modules; the host never invokes `g++` or DALi. `isDockerAvailable()` (`dockerRunner.ts:99‚Äì106`) preflights `docker info` and the render throws if the daemon is unreachable (`:177‚Äì183`), so the host cannot silently fall back to a non-container render. Compilation + rendering occur only inside the image. Upheld.

- **Inv-5 (faithful render = user's real C++ compiled):** **upheld.** `inputResolver.resolveInput` returns the user's verbatim file body (`inputResolver.ts:67‚Äì74`, preview-file mode), and `harnessTemplater` substitutes it *unmodified* into `{{USER_CODE}}` via a literal (non-`$`-interpreting) split-join (`harnessTemplater.ts:79‚Äì81, 118`). The harness then `g++`-compiles+runs that real source in `CreatePreviewUI()` (ADR-002 path; template line 304 walks the *live rendered* `window.GetRootLayer()`), and success requires the user's code to actually compile (`dockerRunner.ts:225` exit-0 + `OK:` gate). No `SBBuildNode`/`RENDER_JSON` reconstruction path is present anywhere in the M0 modules. The PNG and the tree both derive from the compiled real code. Upheld.

- **Inv-6 (JSON stdout is the primary contract; image is on-demand):** **upheld (with one correct M0-scope caveat).** `cli.ts` writes *only* the JSON tree to stdout (`cli.ts:122`, `process.stdout.write(JSON.stringify(tree))`); every diagnostic and the version/usage text aside, all error/diagnostic output goes to `console.error` ‚Üí stderr (`cli.ts:102,126,153`; `dockerRunner`/`templater`/`resolver` headers all state "stdout is reserved for the machine contract" and never write stdout). exec-validation WU-5 confirms "single clean JSON value (924 B); diagnostics on stderr only," so stdout parses cleanly. The invariant's stricter clause ‚Äî "no flag ‚Üí no file writes beyond the working PNG the harness needs" ‚Äî is **partially relaxed in M0 by design, not in violation**: M0 has no bare/flagless invocation (the only command is `<input> --image <path>`, and `--image` is *required* per `parseRenderArgs` `cli.ts:84‚Äì86`), so the harness always writes its working `preview.png` and the CLI always copies it out. The "image is on-demand behind a flag" half of the invariant becomes live in M3 when non-image subcommands (`tree`, bare render) exist; in M0 the *stdout-is-primary-JSON* half is the operative, and upheld one. Flag-gated optionality of the PNG is deferred to M3 (spec.md ¬ß2 defers `watch`/`tree`/`overlay` subcommands).

- **Inv-7 (config echoed = config used):** **deferred ‚Äî not applicable in M0.** Config-echo metadata is an M5/F5.2 feature; M0 ships no config flags beyond `--image`/`--version` (spec.md ¬ß2) and emits no config-echo block. The render *does* use fixed defaults consistently (templater 1024√ó600/`Color::BLACK` ‚Üî dockerRunner `PREVIEW_WIDTH/HEIGHT` 1024√ó600 ‚Äî `harnessTemplater.ts:55‚Äì56`, `dockerRunner.ts:41‚Äì42`), so there is no latent inconsistency to carry forward, but the *echo* the invariant asserts is not yet built. Audit at M5.

- **Inv-8 (id identity survives content/position edits):** **deferred ‚Äî not applicable in M0.** Structural-path id stability is an M2/M4 (ADR-004/F4.2) concern; M0 emits no ids, so there is nothing to be stable. Audit when M2 introduces the structural-path id and M4's tree-diff keys on it.

## Propagation impact

- **Affects M1 (primary):** The spike result (ADR-008) *upgrades* M1's tree-schema input, it does not block it. Concretely M1 must now: **(a)** consume `DumpTree(DUMP_FULL)`'s per-node `role/states/text/type/x,y,w,h/path/children` as a semantic+geometric source; **(b)** add a **control-type ‚Üí role map** because `DumpTree`'s default roles are mostly `"unknown"`/`"redundant object"` (the dali-ui DSL controls set no explicit `ACCESSIBILITY_ROLE` ‚Äî spike caveat); **(c)** **merge** `DumpTree` with the property-walk for fields `DumpTree` omits (user text/background colors, `flexProps` ‚Äî which `CollectActorMetadata` already captures), via a join keyed on actor identity / child-index order; **(d)** **decide the bounds source** ‚Äî `DumpTree`'s `x/y/w/h` is now a viable alternative to `CalculateCurrentScreenExtents` for F1.2 (or a cross-check), and either way M1 must replace the harness's current hand-rolled `parentOrigin/anchor` math (template:133‚Äì135). **Inv-2 is the guardrail on all of this:** whatever M1 builds, the property-walk floor must still yield a complete schema-valid tree with the bridge absent, and a `semanticsSource:"bridge"|"reconstructed"` field must record which path ran (architecture.md Inv-2; ADR-003). M1 now ships *two* sources where the pessimistic plan assumed one ‚Äî slightly more join logic, materially richer output, zero schema-shape change.

- **Affects M2:** `DumpTree`'s per-node bounds (`x/y/w/h`) are confirmed available headless, so they are usable for the Set-of-Mark overlay and coordinate‚Üínode mapping (ADR-008 ¬ßAffected M2) ‚Äî additive, no design change. Inv-1's single-DFS id+mark emission (deferred above) lands here and should be audited then.

- **Affects M5:** The headless run emits a LOUD eldbus/D-Bus stderr deluge (eldbus backtraces, "cannot get dbus connection", spike Evidence A; the M0 dockerRunner already partially silences via `EINA_LOG_*` env ‚Äî `dockerRunner.ts:203‚Äì204`). M5's structured-error contract (`errorParser` g++ ‚Üí `{phase,message,sourceLine}`) **must filter this bridge noise** so it never pollutes the error output (ADR-008 ¬ßAffected M5). `errorParser.ts` is already vendored (vscode-stripped) for this, just not yet wired.

- **Affects M3/M4:** No new propagation from M0 beyond what the ADRs already scope. Note for M3: Inv-6's "image only on a flag" clause (relaxed in M0 because `--image` is mandatory) becomes a real requirement once a bare/`tree` invocation exists ‚Äî the M3 implementer should make PNG production flag-gated then, not assume M0's always-write behaviour is the contract.
